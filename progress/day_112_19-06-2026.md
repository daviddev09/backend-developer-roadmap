# День 112 (19.06.2026): Управление каскадами (Cascades) и «сиротами» (Orphans) в One-to-Many
## Теория:
* Что такое каскадное поведение на уровне бизнес-логики приложения и зачем управлять им через ORM.
* Разбор флагов параметра cascade="..." в функции relationship():
* save-update (включен по умолчанию) — автоматическое добавление связанных объектов в сессию.
* merge (включен по умолчанию) — каскадное слияние состояний объектов.
* delete — если удаляется родитель (User), ORM автоматически сгенерирует SQL-запросы на удаление всех его дочерних записей (Tasks).
* delete-orphan (уничтожение «сирот») — важнейший флаг. Если дочерний объект (Task) удаляется из списка коллекции родителя (user.tasks.remove(task)), он становится «сиротой».Без этого флага объект просто отвяжется (его user_id станет NULL), а с этим флагом — физически удалится из базы данных.
* Архитектурное различие: каскады ORM SQLAlchemy против каскадов СУБД (ON DELETE CASCADE). Почему каскады ORM работают медленнее (так как вызывают объекты в память), но безопаснее для сложных бизнес-правил, и как их правильно комбинировать.
## Практика:
* Настрой на связи tasks в модели User каскадное поведение: cascade="all, delete-orphan".
* Напиши тестовый скрипт, который:
1.	Извлекает пользователя из базы данных вместе с его задачами.
2.	Удаляет одну конкретную задачу из списка user.tasks с помощью стандартных методов работы со списками.
3.	Вызывает session.commit().
* Проверь через логи SQLAlchemy или клиент DBeaver, что СУБД выполнила запрос DELETE FROM tasks WHERE ..., полностью уничтожив отвязанную задачу.
* Сгенерируй Alembic-миграцию, чтобы зафиксировать изменения, и проверь структуру сгенерированного файла ревизии.

## Конспект изученных материалов
### 1. Что такое каскадное поведение в ORM
* **Каскады в ORM** — это правила, которые указывают SQLAlchemy автоматически распространять операции (сохранение, обновление, слияние, удаление), примененные к родительскому объекту (User), на все связанные с ним дочерние объекты (Tasks).
### 2. Разбор флагов параметра cascade='...'
* **A.** **save-update** (включен по умолчанию в cascade="all"): все связанные объекты, которые лежат в списке user.tasks, автоматически добавляются в сессию вместе с ним
* **B.** **merge** (включен по умолчанию в cascade="all"): Когда ты используешь метод session.merge() (например, для синхронизации состояния отсоединенного от сессии объекта с базой), ORM каскадно применит операцию слияния ко всем дочерним элементам в коллекции
* **C.** **delete** (входит в группу all): Когда удаляется родитель из сессии, то перед теи как отправить запрос для удаления users, он идёт и удаляет все его дочерние элементы
* **D.** **delete-orphan** (уничтожение «сирот» — самый важный флаг!): Этот флаг не входит в поведение по умолчанию и прописывается руками: cascade="all, delete-orphan". Когда в ORM ты удаляешь задачу пользователя user.tasks.remove(some_task) то SQLAlchemy понимает что задача стала сиротой и генерирует SQL запрос для удаления этой задачи из БД, а иначе он останется как NULL
### 3. Каскады ORM против Каскадов СУБД
* **Своими словами** : Много писать не ъочу и поэтому лучше комбинировать их ведь так будет безопаснее и на уровни СУБД
