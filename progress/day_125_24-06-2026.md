# День 125: Введение в JWT (JSON Web Token) и Access токены
## Теория: Анатомия токена и Stateless архитектура
* Stateful (Сессии) vs Stateless (Токены): В чем разница подходов. Почему JWT идеально подходит для микросервисов и REST API (серверу не нужно хранить состояние авторизации в оперативной памяти).
* Структура JWT: Глубокий разбор трех частей токена:
* **1.** Header (Заголовок): Тип токена и алгоритм подписи (обычно HS256 или RS256).
* **2.** Payload (Полезная нагрузка): Claims (утверждения). Стандартные claims: sub (subject/user_id), exp (expiration time), iat (issued at). Пользовательские claims (роли, права).
* **3.** Signature (Подпись): Как она формируется и почему злоумышленник не может изменить Payload, не сломав подпись.
* Кодирование Base64Url: Почему токены выглядят как три строки, разделенные точками, и как они легко декодируются (JWT не скрывает данные, он только защищает их от подмены!).
* Секретный ключ (SECRET_KEY): Важность хранения ключа в переменных окружения (.env) и последствия его утечки.
## Практика: Генерация и валидация токенов
* Установите библиотеку PyJWT.
* Добавьте в .env файл секретный ключ (сгенерируйте длинную случайную строку) и укажите алгоритм ALGORITHM = "HS256".
* В файле security.py напишите функцию create_access_token(data: dict) -> str.
* Настройте функцию так, чтобы она принимала ID пользователя, добавляла к нему время жизни токена (например, 15 минут) в поле exp, и подписывала токен секретным ключом.
* Напишите небольшой скрипт для теста: сгенерируйте токен, выведите его в консоль, затем скопируйте и вставьте на сайт jwt.io, чтобы визуально увидеть, как он расшифровывается.

## Конспект изученных материалов
### 1. Сессии против Токены
#### 1. Stateful-архитектура (Сессии/Cookies)
* Исторически первый подход
* **Как работает:** Сервер генерирует случайную строку (Session ID), сохраняет её у себя в оперативной памяти или в базе данных, и отправляет клиенту. Клиент присылает Session ID с каждым запросом
* **Проблема:** Если есть миллион активных пользователей, серверу нужно хранить миллион Session ID в памяти. А если серверов несколько (микросервисы), им придется постоянно синхронизировать этот "список гостей" между собой. Это тяжело масштабировать.
#### 2. Stateless-архитектура (JWT Токены)
* Современный подход для REST API. Stateless означает "Без сохранения состояния"
* **Как рабоатет:** При логине сервер собирает небольшой JSON с данными пользователя, математически подписывает его своим секретным ключом и отдает клиенту. Клиент присылает этот токен с каждым запросом. Сервер берет токен, проверяет криптографическую подпись и сразу понимает: "Ага, подпись моя, значит внутри достоверные данные, это юзер №1"
### 2. Структура JWT
* В JWT есть три составные части: Header(Заголовок), Payload(Полезная нагрузка) и Signature(Подпись).
#### 1. Header (Заголовок) ХХХХХ
* Обычный JSON, который содержит метаданные о самом токене, например: {"alg": "HS256", "typ": "JWT"} Он говорит: " Я - токен типа JWT, и моя подпись сгенерирована алгоритмом HMAC SHA-256".
#### 2. Payload (Полезная нагрузка/Claims) yyyyy
* Здесь хранятся сами данные. В терминах JWT эти данные называются *Claims* (Утверждения). Есть стандартеые **Claims** (зарезервированные слова):
* **sub** (Subject): Кому принадлежит токен (Обычно сюда кладут user_id)
* **exp** (Expiration time): Время смерти токена в формате Unix Timestamp. По стечении этого времени сервер отвергнет токен
* **iat** (Issued at): Время выдачи токена
* Есть пользовательские **Claims** и туда можно положить что угодно, например, роль пользователя 
#### 3. Signature (Подпись) zzzzz
* Это самая важная часть. Она гарантирует, что хакер не изменил Payload. Если не знать ключ и попытаться изменить Payload сервер сразу поймёт что токен был изменён и выдаст ошибку 401 Unautorized
### 3. Кодирование Base64Url
* В JWT Header и Payload не зашифрованы. Они просто закадированы в формат Base64Url, чтобы безопасно передаваться в HTTP-заголовках. Любой человек может взять JWT вставить его в декодер и прочитать Payload в виде обычного текста, и поэтому в Payload никогда нельзя класть пароли, номера крединтых карт или секретные ключи.
### 4. Секретный ключ (SECRET KEY)
* Вся надежность JWT держится на одном SECRET KEY
* Это обязательно должно быть длинной случайной криптографической строкой, иначе его можно найти методом перебора (brute-force)
* Обязана храниться в .env и никогда не попадать в публичный репозиторий
* Если хакер украдет SECRET KEY, он сможет подписывать им свои токены, сгенерировав токен {"role":"super_admin"} полностью захватить приложение
