# День 155: Основы Rebase (Перебазирование)
## Теория:
* Отличия rebase от merge и зачем нужна линейная история.
* Как rebase переписывает историю, создавая новые хэши коммитов.
* Золотое правило Rebase: Никогда не делай rebase для публичных веток (например, main), с которыми работают другие люди!
Команды для работы: git rebase main (находясь в своей ветке) — перенести свои изменения поверх main; git rebase --continue — продолжить после решения конфликтов; git rebase --abort — отменить rebase.
## Практика:
* Создай ветку feature-3 от main, сделай коммит.
* Вернись в main, сделай другой коммит (эмулируя работу коллег).
* Перейди в feature-3 и выполни git rebase main.
* Посмотри историю через git log --graph — она должна стать линейной, без merge-коммита.

## Конспект изученных материалов
### 1. Отличия rebase от merge
* Когда используется классический merge, Git сохраняет всю историю изменений "как есть". Если история разделилась, слияние через merge создаст специальный merge-коммит, связывающий две разошедшиеся истории. Со временем, если разработчиков много, граф истории начинает быть запутанным.
* В свою очередь rebase берет коммиты из одной ветки и "пересаживает" их поверх другой, создавая красивую линейную историю. Это будет идельно ровная прямая линия, где коммиты идут строго один за другим. Это колоссально упрощает чтение истории проекта: при просмотре логов сразу станет понятно, кто, что и в каком порядке делал
### 2. Как rebase переписывает историю
* Самое важное в механике этого процесса: rebase переписывает историю, создавая новые хэши коммитов.
* Git не просто переносит старые коммиты, он берет суть изменений из каждого коммита в ветке feature и по очереди применяет их поверх последнего свежего коммита в ветке main. В результате в истории появляются абсолютно новые коммиты с новой датой и новым хэшем, хотя внутри сам код и текст сообщения остаются прежними.
### 3. Золотое правило Rebase
* **Никогда не делать Rebase для публичных веток, с которыми работают другие люди!**
* **Это критично, но почему?** Потому что Rebase меняет хэши коммитов. Если сделать rebase общей ветки main, которую уже скачали коллеги, и отправить её на сервер, случится катострофа. У команда на компьютерах останутся старые коммиты со старыми хэшами, а на сервере будет лежать те же самые изменения, но с новыми хэшами. При следующей попытке коллег подтянуть обновления, Git попытается слить две разные истории воедино. Проект мгновенно обрастет дублирующимися коммитами и сотнями конфликтов, а коллеги буду очень недовольны.
* **Где использовать:** нужно применять rebase исключительно к новым личным локальным веткам перед тем, как отправить код на код-ревью. Это поможет "причесать" твою работу и влить её в проект аккуратной линией.
### Команды Git
* *git rebase <имя файла>:* Применяется чтобы перебазировать работу в ветке, например feature-3 на ветку main. Чтобы тспользовать нужно находится в той ветке, которую хотите перебазировать, если хотите перебазировать на main, то команда будет: *git rebase main*
* В процессе rebase Git применяет коммиты по одному, и если возникает конфликт, нужно вручную разрешать конфликты, то есть открыть файлы с конфликтами, устранить их вручную, проиндексировать изменения и продолжить:
    * *git add path/to/conflicted-file.py* добавить файл в индекс
    * *git rebase --continue* продолжить процесс rebase
* *git rebase --abort* если запутался в конфликтах или понял что начал rebase не той ветки, возвращает всё в исходное состояние, существовашее до ввода команды git rebase
* *git rebase --skip* если конфликтующий коммит из ветки больше не нужен, его можно пропустить
