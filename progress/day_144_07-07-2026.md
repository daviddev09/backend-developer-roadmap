# День 144 (07.07.2026): Связывание слоев — Dependency Injection (Внедрение зависимостей)
## Теория:
* Проблема ручного создания объектов: если Роутеру нужен Сервис, а Сервису нужен Репозиторий, а Репозиторию нужна Сессия БД, как не писать этот длинный код инициализации в каждом эндпоинте.
* Мощь системы Depends в FastAPI: использование генераторов для автоматического построения графа зависимостей.
* Жизненный цикл зависимостей: создание сессии БД на время одного HTTP-запроса, проброс её по цепочке и гарантированное закрытие.
## Практика:
* Создай файл api/dependencies.py.
* Напиши функцию get_db(), которая генерирует AsyncSession (с использованием yield).
* Напиши функцию get_user_repository(db: AsyncSession = Depends(get_db)).
* Напиши функцию get_user_service(repo: UserRepository = Depends(get_user_repository)).
* Отрефактори роутер из Дня 143: теперь эндпоинт должен просто принимать service: UserService = Depends(get_user_service) и мгновенно использовать его.

## Конспект изученных материалов
### 1. Проблема ручного создания объектов
* **Своими словами:** Если в каждом эндпоинте: Вручную открывать сессию, вручную создавать объект репозитория, вручную создавать сервис, то когда в проекте будет даже 50 эндпоинтов, это уже 150 +- лишних строк кода, а ещё есть человеческий фактор, случайно можно где-то написать неправильно. А чтение такого кода будет сложным. И чтобы не писать такой длинный код инициализации каждый раз нужно использовать Внедрение Зависимостей
### 2. Мощь Depends
* **Что такое граф зависимостей:** это дерево объектов, например: Router - Service - Repository - AsyncSession. Depends сначала создаёт сессию и в конце вызывает Router. Это называется грфом потому что зависимостей может быть несколько, например Сервис может зависеть сразу от: Репозитория, логгера, Редис и настройек. А FastAPI умеет автоматически строить даже очень сложные графы
### 3. Жизненный цикл зависимостей
* **Этого Я сам далеко не понял:** Жизненный цикл зависимостей это когда приходит HTTP-запрос: FastAPI открывает сессию, создаёт репозиторий, создаёт сервис и вызывает роутер, потом роутер отвечает клиенту и сессия закрывается. И здесь сессия живёт только во время запроса. Думаю это и есть жизненный цикл засисимостей. В общем, жизненный цикл зависимостей это ровно то, насколько живёт сам запрос. Ведь, когда приходит запрос FastAPI создаёт зависимости а когда роутер возвращает ответ, FastAPI уничтожает эти зависимости.
