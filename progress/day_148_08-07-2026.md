# День 148 (08.07.2026): Ручное кэширование в слое Services и Repositories
## Теория:
* Ограничения декораторов: почему кэшировать весь HTTP-ответ не всегда выгодно (например, если ответ зависит от роли пользователя или содержит динамические токены).
* Внедрение Redis в слоистую архитектуру: как правильно передать Redis-клиент через систему Depends прямо в слой бизнес-логики (Services).
* Сериализация моделей: как конвертировать объекты SQLAlchemy или схемы Pydantic в строки (JSON) для сохранения в Redis, и как их правильно десериализовать обратно при чтении.
* Проблема Cache Stampede (Лавина запросов): что произойдет, если кэш популярного товара истечет ровно в тот момент, когда на сайт зашли 1000 пользователей (все 1000 запросов одновременно ударят в базу данных).
## Практика:
* Добавь зависимость получения Redis-клиента в dependencies.py.
* Внедри Redis в UserService.
* Напиши логику в методе получения пользователя по ID: сервис должен самостоятельно генерировать ключ кэша (например, user:profile:{user_id}), проверять его в Redis, и только при отсутствии вызывать метод репозитория, а затем сохранять результат в Redis через команду SETEX.

## Конспект изученных материалов
### 1. Ограничение декораторов
* **Пробелема 1:** Если есть эндпоинт GET/users/me который возвращает профиль и на нём стоит декоратор @cache, первый авторизованный пользователь вызовет его, и ответ закэшируется: {'name': 'Daviddev'}. Когда через секунду на этот эндпоинт зайдет кто-то другой, он получит мой проыфоль.
* **Проблема 2:** Если в ответе возвращается CSRF-токен, который должен быть уникальным для каждого запроса кэшировать ответ целиком нельзя - токен устареет, и фронтенд сломается
* **Проблема 3:** Обычный пользователь может видеть одни полня, а админ может другие.
* **Вывод:** Роутер должен принимать только HTTP-запрос, а кэшированием должна заниматься бизнес-логика.
### 2. Внедрение Redis в слоистую архитектуру
* **Своими словами:** Redis нужно внедрать через Dependency Injection, в файле dependencies.py надо создать функцию-генератор get_redis(). Потом инжектить этот Redis-клиент в Сервис.
### 3. Сериализация и дисериализация моделей
* **Сериализация** это процесс превращения Python объекта в JSON строку
* **Десериализация** это процесс превращения JSON строки в Python объект
* Redis умеет хранить только байты и строки. И поэтому в него нельзя передавать объекты ORM или Pydantic. **Сериализация** берётся объект из базы и прогоняется через DTO и превращается в JSON-строку с помощью метода .model_dump_json(). Эта строка пишется в Redis. **Десериализация** Читается строка из Redis. Чтобы можно было работать с ней как с объектом а не словарем, нужно скормить эту строку Pydantic-схеме с помощью метода .model_validate_json(raw_string).
### 4. Проблема Cache Stampede
* **Например:** Есть интернет магазин. На главной странице висит востребованный товар со скидкой в 90%. Страницу открывают 1000 человек в секунду. Данные в кэше Redis с TTL=60. Когда таймер станет 0, кэш удаляется из памяти. И ровно в эту же миллисекунду приходят 1000 новых запросов от пользователей. 
* * **1.** Все 1000 потоков FastAPI одновременно спрашивают Redis: Дай данные товара
* * **2.** Redis всем потокам отвечает: Кэша нет
* * **3.** Все 1000 потоков одновременно идут выполнять тяжелый SQL-запрос к Postgres
* * **4.** Postgres пытается одновременно выполнить 1000 тяжелых джоинов, его процессор уходит в 100%, и БД умирает
* **Как с этим борются:** Чаще всего используют механизм Блокировки. Когда кэш протухает, первый запрос, который увидел Miss, ставит в Redis специальный флаг-замок (Я пошел в базу остадьные ждите). Остальные 999 запросов видят замок и просто ждут, после чего снова проверяют кэш. Первый запрос приносит данные из БД, положит их в кэш, снимет замок, и остальные 999 мгновенно получат готовый результат, не потревожив БД
* **Есть и другие методы борьбы: случайный TTL, предварительное обновление кэша и другие стратегии
