# День 132: Архитектура очередей задач (Celery + Redis)
## Теория:
* Концепция распределенных очередей задач (Task Queues): как вынести тяжелую работу из веб-сервера (FastAPI) в совершенно независимые процессы.
* Роль Message Broker (Брокера сообщений): зачем нам нужны инструменты вроде Redis или RabbitMQ. Как брокер работает в качестве "почтового отделения", принимая задачи от FastAPI и раздавая их воркерам.
* Роль Workers (Воркеров): независимые запущенные процессы, которые слушают брокер, забирают задачи и выполняют их.
* Библиотека Celery: абсолютный индустриальный стандарт в Python для построения надежных фоновых задач с гарантией выполнения, повторными попытками при ошибках (retries) и расписаниями (cron).
## Практика:
* Установи Redis на свой компьютер (быстрее и чище всего сделать это через Docker командой docker run -p 6379:6379 -d redis).
* Установи библиотеку celery в свое виртуальное окружение.
* Создай файл worker.py и инициализируй в нем базовый экземпляр приложения Celery, подключив его к локальному Redis.
* Напиши свою первую задачу: функцию сложения двух чисел, обернутую в декоратор @celery.task.

## Конспект изученных материалов
### 1. Концепция распределённых очередей задача
* **Проблема встроенных фоновых задач:** Они выполняются прямо внутри оперативной памяти веб-сервера. Если сервер упадет - задача исчезнет. Если 1000 пользователей одновременно решат сгенерировать тяжёлый PDF-отчет, сервер будет загружен на 100%. FastAPI перестанет отвечать на новые запросы.
* Решение: Вынести работу наружу** распределенная очередь задач, это архитектурный паттерн, при котором тяжелая работа физически выноситься за пределы веб-сервера. FastAPI должен только сказать: "нужно сгенерировать отчет для юзера №5", и тут же забыть об этом, моментально вернув ответ клиенту. А саму генерацию отчета будет делать совершенно другая программа, возможно, даже запущенная на другом компьютере. **Для работы этой магии нужны две сущности: Message Broker и Worker 
### 2. Роль Message Broker
* **Брокер сообщений** это сверхбыстрое специализированное хранилище (часто работающее в оперативной памяти)
* **Аналогия** Это как лента для чеков на кухне ресторана. Официант (FastAPI) принимает заказ у клиента, пишет его на бумажке и вешает на ленту. Официант не идет готовить суп, он идет к следующему клиенту. Брокер (лента) просто надежно хранит этот чек, пока его кто-нибудь не заберет.
### 3. Роль Workers
* **Воркеры** это "повара" на кухне.
* Это абсолютно независимые процессы Python. Они не знают ничего про HTTP, про эндпоинты или роутеры. У них есть только одна цель:
* **1** Подключиться к Redis
* **2** Спросить: "Есть новые задачи?""
* **3** Если задача есть - забрать её (чек снимается с ленты чтобы другой воркер его не взял)
* **4** Выполнить тяжёлую работу (скачать файл, обучит нейросеть и т.д.)
* **5** Повторить цикл
* **Главный плюс архитектуры:** Масштабируемость. Если перед каким-то событием нагрузка на сайт вырастет в 10 раз, то нам не нужно трогать код FastAPI. Нужно просто запустить в турминале не 1 воркер, а 50 воркеров. Они все подключаться к одному Redis и начнут разгребать очередь задач с бешеной скоростью
### 4. Библиотека Celery
* Чтобы FastAPI, Redis и Воркеры понимали друг друга, нужен общий язык. Нам не нужно вручную писать код для сериализации задач JSON и складывания их в Redis. Для этого есть Celery.
* **Celery** - это мощнейшая библиотека, которая берет на себя всю "грязную" работу:
* **1** Управляет подключением к брокеру (Redis)
* **2** Сама упаковывает аргументы функции и отправляет их в очередь
* **3** Сама запускает процессы-воркеры
* **4** Гарантия доставки: Если воркер начал делать задачу, но у него отключился ток, Celery поймет, что задача не заврешена, и вернет её обратно в Redis, чтобы её сделал другой воркер!
* **5** Retries(Повторы): Можно сказать Celery: "Если FastAPI стороннего банка не отвечает, попробуй выполнить задачу ещё 3 раза с интервалом в 10 минут"
