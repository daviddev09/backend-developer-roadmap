# День 141: Слой доступа к данным — Паттерн Repository
## Теория:
* Проблема прямого использования session.execute() в бизнес-логике: жесткая привязка всего кода к SQLAlchemy.
* Паттерн Repository (Репозиторий): создание класса-прослойки, который инкапсулирует в себе все SQL-запросы.
* Изоляция базы данных: почему Репозиторий должен принимать простые типы Python (или Pydantic-схемы) и возвращать ORM-модели, не выпуская наружу специфичные исключения базы данных.
* Базовый класс репозитория (Generic/Base Repository): как использовать дженерики Python (TypeVar), чтобы написать методы get_by_id, create, update один раз и наследовать их для всех таблиц проекта.
## Практика:
* Создай файл repositories/base.py и напиши класс BaseRepository, который принимает AsyncSession при инициализации.
* Реализуй в нем базовые асинхронные CRUD-методы, используя SQLAlchemy 2.0.
* Создай файл repositories/user.py, унаследуй UserRepository от базового и добавь туда специфичный метод get_user_by_email(email: str).

## Конспект изученных материалов
### 1. Проблема обращения к БД в бизнес-логике
* **1. Жётская привязка (Tight Coupling):** Если UserService знает всё про функции SQLAlchemy, когда кто-то решит изменить SQLAlchemy на другую ORM, то придется переписать весь бизнес-слой
* **2. Дублирование кода:** Если нужно получить какую-то сущность по ID в UserService, потом в PostService а потом ещё и в TaskService, придется писать один и тотже select трижды
* **3. Невозможность тестирования:** Чтобы протестировать бизнес-логику, обязательно придется поднимать реальную БД. Без Репозитория невозможно легко подменить БД
### 2. Паттерн Repository
* **Своими словами:** Паттерн Репозиторий это про написание отдельного слоя приложения которое служит прослойкой между бизнес-логикой и БД. Сервис в таком случае не знает ничего о том: Какой ORM или какая БД используется.
### 3. Изоляция БД
* **Своими словами:** Репозиторий должен перехватывать ошибки SQL и выкидывать понятные ошибки Python иначе, PostgreSQL может выдать ошибку: UniqueViolation и если эта ошибка выходит за пределы Репозитория, то программа узнает что существует Postgres, SQLAlchemy, Unique, а это нарушение архитектуры. Репозиторий должен принимать базовые типы, потому что он сам с них должен строить SQL-запросы и возвращать ORM-модели ведь именно с ними будет работать сервис
### 4. Базовый Репозиторий
* **Своими словами:** Если в проекте, допустим 10 таблиц, то в каждом репозитории: UserRepository, TaskRepository..., мы будем писать одни и те же методы get_by_id, create, update и delete это очень много. Поэтому в Python надо использовать Generics и TypeVar, чтобы написать базовый класс Репозитория и наследовать его для других репозиториев.
### Что такое TypeVar
* TypeVar означает **переменная типа**
* **Она говорит:** Я не знаю заранее, с каким типом будет работать этот класс. Пусть пользователь подставит любой.
### Что делает *bound=Base*?
* **Вот эта строка:** *ModelType = TypeVar('ModelType', bound=Base)* означает: "ModelType может быть только Base или наследником Base", то есть всё(int, str, dict...) кроме него присваивать туда запрещается
### Что такое Generic:
* Вот это: *"class BaseRepository(Generic[ModelType]):"* говорит: "Этот класс является универсальным", то есть один и тот же класс можно использовать с разными моделями.
* **Например:** если *repo = BaseRepository(User, session)*, то *Modeltype = User*. А если *repo = BaseRepository(Product, session)* то *Modeltype = Product*
### Что такое ModelType:
* **После:** *ModelType = TypeVar('ModelType', bound=Base)*, ModelType становится просто именем типа. Можно представить, что это неизвестная модель, он может быть: *ModelType=User* или *=Product* или *=Order*. Когда IDE видит ModelType она понимает: "ага, здесь будет конкретная ORM-модель" ведь *bound=Base* а это значит там может быть Base или его наследники
