# День 122: Автоматическое тестирование с Pytest
## Теория:
* Пирамида тестирования: разница между Unit-тестами (модульными) и Integration-тестами (интеграционными).
* Библиотека pytest: синтаксис, структура тестовых файлов (префикс test_) и запуск тестов из консоли.
* Инструмент TestClient из библиотеки httpx (встроен в FastAPI): как симулировать HTTP-запросы к твоему API без реального запуска сервера.
* Фикстуры (@pytest.fixture): мощный механизм для подготовки данных перед тестом (например, создание пустой тестовой базы данных и её очистка после завершения теста).
* Переопределение зависимостей (app.dependency_overrides): как подменить боевую сессию базы данных (get_db) на тестовую во время прогона тестов.
## Практика:
* Установи пакеты pytest и httpx в свое виртуальное окружение.
* Создай папку tests/, а внутри нее файл test_posts.py.
* Напиши фикстуру, которая будет создавать тестовую SQLite-базу (или отдельную схему в Postgres) специально для тестов.
* Напиши тест для эндпоинта создания поста (POST /posts/): отправь тестовый JSON через TestClient, проверь, что статус-код равен 201, а в ответе вернулся правильный title
* Напиши тест для получения списка постов (GET /posts/), проверь корректность структуры данных в ответе.

## Конспект изученных материалов
### 1. Пирамида тестирования
* В бэкенде тесты деляться на уровни (Пирамида):
* **Unit-тесты:** Проверяют крошечные изолированные куски кода, например функцию которая хеширует пароль. 
* **Integration-тесты:** Они проверяют то, как компоненты работают вместе. Для API интеграционные тесты самые важные
### 2. Библиотека pytest
* **pytest** - Это мощный фреймворк для тестирования
* **Имена файлов:** pytest ищет только те файлы которые написаны как: test_ например, test_posts.py или же те которые заканчиваются на _test
* **Внутри файлов** тестом считается только те функции которые названы test_ например, async def test_create_user()
### 3. Инструмент TestClient
* **Как тестировать данные?** Внутри тестов нам надо запускать сервер, открывать браузер и тыкать в Swagger? Нет! Это долго и нестабильно
* В FastAPI встроен TestClient 
* **Он** не открывает реальный порт (например, 8000)
* **Он** шлет запросы напрямую в ASGI-Interface твоего приложения
### 4. Фикстуры и yield
* **Фикстуры** - это помощники которые готовят сцену для теста и убирают мусор после него
* **Например** перед тем как запуститть тесты, нам нужно подготовить фейковые таблицы в БД. А после тестов - удалить их, чтобы следующий запуск тестов проходил на чистой базе
### 5. Переопределения зависимостей (dependency_overrides) 
* Это самая важная концепция для БД в FastAPI
* Боевой эндпоинт GET/users/ зависит от функции get_db() которая подключается к БД если запустить их так, то тесты накидают мусора в настоящую БД
* Но нужно перенаправить тесты в отдельную базу (например, fastapi_test_db). Для этого в FastAPI есть словарь app.dependency_overrides. Мы говорим приложению: "Эй, если кто-то попросит зависимость get_db, дай ему вместо нее функцию override_get_db".
