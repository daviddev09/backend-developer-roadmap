# День 146 (08.07.2026): In-Memory базы данных и фундамент Redis
## Теория:
* Физика кэширования: почему чтение из оперативной памяти (RAM) занимает наносекунды, а чтение с жесткого диска (даже SSD) — миллисекунды. Зачем нужен кэш между FastAPI и PostgreSQL.
* Концепция Key-Value хранилищ: чем Redis радикально отличается от реляционных баз данных. Отсутствие таблиц, схем и связей.
* Структуры данных в Redis: изучение работы со строками (Strings), списками (Lists), множествами (Sets) и хэшами (Hashes).
* Механизм TTL (Time To Live): как работает автоматическое удаление устаревших данных (команда EXPIRE) и почему это критически важно для предотвращения переполнения оперативной памяти.
* Политики вытеснения (Eviction Policies): что делает Redis, когда память заканчивается (разбор алгоритма LRU — Least Recently Used).
## Практика:
* Запусти Redis локально через Docker (если еще не запущен с 132-го дня).
* Подключись к нему через консоль redis-cli и отработай базовые команды: SET, GET, DEL, EXPIRE, TTL.
* Установи библиотеку redis (или redis.asyncio) в свой Python-проект.
* Напиши изолированный скрипт, который асинхронно подключается к Redis, записывает сложный словарь (предварительно сериализовав его в JSON), устанавливает время жизни в 10 секунд и затем читает его обратно.

## Конспект изученных материалов
### 1. Физика кэширования
* **Своими словами:** RAM это очень быстрая временная память которые хранит данные пока работает, а SSD или HDD это диск который хранит данные навсегда и поэтому они медленне чем RAM.
* **Кэш между FastAPI и Postgres:** Если один пользователь открывает главную страницу сайта, делает запрос к БД, а это занимает миллисекунды и после этого ещё тысячи пользователей открывают главную страницу сайта - итог, тысяча одинаковых запросов которые попросту убивают производительность. **Решение:** Кэшировать данные в RAM чтобы когда поступали одинаковые запросы данные брались из памяти а не из БД так как это очень быстро.
### 2. Концепция Key-Value хранилищ
* **Redis:** это key-value storage, здесь данные хранятся в виде: ключ: значение. И этим он отличается от Postgres, ведь у него нету сложных структур таблиц, связей, транзакция и т.п. и поэтому он быстрый. Ключ здесь это название записи, а значение это сама запись, ключом может быть id пользователя а значением, его профильные данные
### 3. Структуры данных в Redis
* **Своими словами:** В Redis значение (Value) может быть разных типов. Например:
* * **Strings:** Базовый тип, занчением может быть обычный текст, JSON строка, число или бинарный файл. Команды: *SET key value*, *GET key*
* * **Lists:** Массивы строк. Идеально для создания простых очередей задач или хранения "последних 10 действий пользователя". Команды: *LPUSH* (добавить слева), RPOP (забрать справа).
* * **Sets:** Коллекция уникальных строк (как set в Python). Идеально для учета уникальных посетителей (один IP добавится только один раз) или системы тегов. Команды: *SADD key value* (добавить), *SISMEMBER key value* (проверить наличие).
* * **Hashes:** Это вложенные словари. Значение ключа является набором полей и значений. Идеально для хранения корзины интернет-магазина или сессии конкретного пользователя (где поля - это username, role, theme)
### 4. Механизм TTL
* **Своими словами:** Механиз TTL (Time To Live) - это время жизни кэша. И по истечению своего времени Redis удалит этот кэш из памяти автоматически. Команда: *EXPIRE key 60* устанавливает таймер на существующий ключ, *SETEX key 60 value* записывает значение и сразу ставит таймер, это - *SET* + *EXPIRE*
#### 5. Политики вытеснения
* **Своими словами:** Это алгоритм который вытесняет (удаляет) данные из кэша чтобы освободить место когда память начинает переполняться. Самый известный алгоритм это LRU - Least Recently Used (Наименее недавно использованный). Когда память заполнена 100 и надо добавить новые данные в кэш, Redis использует этот алгоритм и смотрит на данные и находит самые часто неиспользуемые данные и удаляет их
