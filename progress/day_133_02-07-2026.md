# День 133: Интеграция Celery с FastAPI и мониторинг статусов
## Теория:
* Запуск задач из FastAPI: использование методов .delay() и .apply_async() для отправки задачи в брокер без блокировки веб-сервера.
* Жизненный цикл Celery-задачи: разбор статусов PENDING (в очереди), STARTED (выполняется), SUCCESS (успех), FAILURE (ошибка).
* Концепция Backend (Хранилище результатов): как настроить Celery на сохранение результатов выполнения задачи обратно в Redis, чтобы FastAPI мог проверить их позже по уникальному task_id.
* Паттерн Polling (Опрос): как клиент (фронтенд) должен периодически запрашивать у бэкенда статус своей долгой задачи, используя её ID.
## Практика:
* Запусти воркер Celery в отдельном окне терминала (например, celery -A worker.celery worker --loglevel=info).
* В FastAPI создай POST-эндпоинт /reports/, который вызывает тяжелую Celery-задачу генерации отчета через .delay(). Эндпоинт должен мгновенно вернуть клиенту JSON с уникальным task_id.
* Создай GET-эндпоинт /reports/{task_id}, который обращается к Celery, проверяет текущий статус задачи по этому ID и возвращает его клиенту.
* Запусти генерацию отчета и несколько раз дерни GET-эндпоинт, чтобы увидеть, как статус меняется с PENDING на SUCCESS.

## Конспект изученных материалов
### 1. Запуск задач из FastAPI
* Мы не можем просто вызвать синхронную функцию внутри эндпоинта FastAPI так как это заблокирует сервер. Нам нужно передать эту задачу в Celery. Для этого объект задачи, созданный через декоратор @celery_app.task, имеет специальные методы
* **А.** Метод .delay(*args, kwargs) это самый простой и популярный способ. Он мгновенно отправляет задачу в очередь со стандартными настройками
* **B.** Метод .apply_async(args=[], kwargs={}, options) Это продвинутый способ. Он позволяет настроить, как именно и когда должна выполниться задача. Например, мы можем запустить задачу не прямо сейчас, а через 60 секунд
* Оба метода возвращают объект AsyncResult
### 2. Жизненный цикл Celery задачи
* Когда задача попадает в Celery она проходит через несколько состояний.
* **PENDING** (В ожидании): задача находиться в очереди и ждет, когда освободиться воркер
* **STARTED** (Начата): Воркер забрал задачу из очереди и начал её выполнять
* **SUCCESS** (Успех): Задача выполнилась успешно
* **FAILURE** (Ошибка): Во время выполнения функции возникло исключение. Результатом задачи в этом случае будет само исключение с трассировкой
### 3. Концепция Backend
* Если у нас настроен только брокер, воркер просто выполнит задачу, выведет логи в консоль и забудет о ней. FastAPI никогда не узнает, успешно ли выполнилась задача и какой результат она вернула
* Чтобы FastAPI мог проверить статус (Успех/Ошибка) и забрать результат, надо настроить Result Backend. Чаще всего в качестве Backend`a используют тот же самый Redis 
### 4. Паттерн Polling
* Чтобы передать резултат клиенту, когда HTTP-запрос закрывается мгновенно, используется классический архитектурный паттерн Polling
* **Как это работает**
* **1.** Пользователь на фронтенде нажимает на кнопку: Сгенерировать тяжелый отчет
* **2.** Фронтенд отправляет POST-запрос к FastAPI
* **3.** FastAPI отправляет задачу в Celery через .delay() и мгновенно возвращает клиенту JSON с task_id
* **4.** Фронтенд показывает пользователю "Лоадер" и начинает с помощью setInterval делать GET-запрос на эндпоинт статуса передавая туда task_id
* **5.** FastAPI через Celery проверяет Backend. Если статус PENDING, он возвращает фронтенду "Подожди ещё" 
* **6.** Как только FastAPI видит статус SUCCESS, он достает результат и отдает его фронтенду.
* **7.** Фронтенд останавливает таймер, прячет "Лоадер" и показывает пользователю ссылку на скачивание.
