# День 151 (12.07.2026): Автоматизированное тестирование (Pytest) и Mocking
## Теория:
* Пирамида тестирования: разница между Unit-тестами (тестирование изолированных функций/сервисов), Integration-тестами (связка эндпоинта, сервиса и тестовой базы данных) и E2E (сквозное тестирование).
* Философия Pytest: почему pytest стал абсолютным стандартом в Python. Понимание механизма assert (утверждений) и того, как Pytest перехватывает ошибки.
* Фикстуры (Fixtures): мощнейший инструмент Pytest для подготовки состояния перед тестом. Как с помощью декоратора @pytest.fixture создавать тестового пользователя, открывать соединение с БД и автоматически удалять данные после завершения теста.
* Тестирование FastAPI: класс TestClient (из starlette.testclient) для синхронных тестов и httpx.AsyncClient для полноценного асинхронного тестирования эндпоинтов.
* Переопределение зависимостей (Dependency Overrides): важнейший паттерн FastAPI. Как с помощью app.dependency_overrides подменить реальную боевую базу данных на изолированную тестовую базу (например, на отдельную схему PostgreSQL или In-Memory SQLite) только на время выполнения тестов.
* Mocking (Имитация): почему юнит-тесты не должны отправлять реальные письма и стучаться в реальный Redis. Изучение библиотеки unittest.mock (и pytest-mock) для создания заглушек (patching).
## Практика:
* Установи пакеты pytest, pytest-asyncio и httpx в свое виртуальное окружение.
* Создай в корне проекта директорию tests/, а внутри файл conftest.py (специальный файл Pytest, где хранятся глобальные фикстуры).
* Напиши фикстуру test_db, которая создает пустую тестовую базу данных перед запуском тестов и удаляет таблицы после их завершения.
* Напиши фикстуру async_client, которая инициализирует httpx.AsyncClient и подключается к твоему приложению FastAPI.
* Создай файл test_auth.py и напиши два интеграционных теста:
* Тест успешной регистрации: отправка POST-запроса на /register с валидным JSON, проверка статуса 201 и наличия email в ответе.
* Тест ошибки при дублировании: попытка зарегистрировать пользователя с тем же email второй раз и проверка того, что сервер корректно возвращает HTTP-статус 409 (Conflict).
* Запусти тесты командой pytest -v и добейся зеленого вывода в консоли.

## Конспект изученных материалов
### 1. Пирамида тестирования
* **Пирамида тестирования состоит из трёх частей:**
    * **Unit-тесты:** Проверяют крошечные, изолированные куски кода (Например, функцию хэширования пароля). Они не запускают FastAPI и не ходят в БД
    * **Inegration-тесты:** Это то, что пишется для API. Они проверяют как компоненты работают вместе: роутер принял запрос - вызвал сервис - сервис дернул репозиторий - репозиторий сходил в тестовую БД
    * **E2E (Сквозные):** Тестирование всей системы от лица пользователя, обычно делают QA-инженеры
### 2. Философия Pytest
* **pytest** стал абсолютным стандартом благодаря своей простоте. И чтобы использовать его не нужно писать сложные методы вроде: *assertEqual(a, b)*, нужно просто написать *assert a == b*
### 3. Фикстуры
* **Фикстура** это функция-помощник, которая готовит окружение перед тестом. Например, перед тем как тестировать эндпоинт регистрации, нам нужна чистая база данных. Фикстура может:
    * Создать таблицы (Setup)
    * Заморозить свое выполнение через оператор yield и передать управление тесту
    * Дождаться окончания теста и удалить таблицы (Teardown)
### 4. Тестирование асинхронного FastAPI
* Для тестирования HTTP-запросов используется библиотека **httpx**, у нее есть класс **AsyncClient** который позволяет отправлять запросы прямо в FastAPI приложение, не открывая реальный порт 8000. Это делает тесты невероятно быстрыми
### 5. Переопределение зависимостей
* Это важнейший паттерн, ведь в коде есть зависимость *get_db_session* которая возвращает реальную БД. Но если использовать его для тестов они испортят реальную БД мусором или удалят нужные данные. FastAPI позволяет временно подменить зависимость: *app.dependency_overrides[get_db_session] = override_get_db* 
### 6. Mocking
* Если при регистрации пользователя, код отправляет реальное письмо через SMTP или обращается к Celery/Redis, тесты начнут тормозить, рассылать спам или падать из-за отсутствия интернета. Библиотека unittest.mock позволяет создать "заглушку". И эта заглушка когда вызывается функция *send_email* просто не выполняет её а притворяется что она вернула True
### Объяснение функций
* **httpx.AsyncClient:** Это аналог стандартного *requests.Session()* или *httpx.Client()*. Предназначен для отправки асинхронных HTTP-запросов (GET, POST и т.д.) без блокировки Event Loop. В тестах используется чтобы отпарвлять асинхронные запросы к FastAPI
* **httpx.ASGITransport:** Это специальный механизм ("Транспорт"), который позволяет *httpx.AsyncClient* общаться с ASGI-приложением, то есть с FastAPI напрямую в памяти процессора, вообще не поднимая сетевой веб-сервер. Нужен в основном для тестирования
* **abc.AsyncGenerator:** Это тип данных описывающий асинхронный генератор.
* **Краткая архитектурная взаимосвязь:**
    * **1.** Пишутся тесты для Auth-сервиса.
    * **2.** С помощью *AsyncGenerator* настраивается тестовая фикстура, которая создает и закрывает клиент.
    * **3.** В этот клиент внедряется *ASGITransport*, чтобы запросы шли напрямую в код приложения без сети
    * **4.** Сам код тестов использует *AsyncClient* для выполнения асинхронных вызовов эндпоинтов.
