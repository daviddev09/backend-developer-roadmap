# День 152 (13.07.2026-16.07.2026): OpenAPI (Swagger), CORS, Конфигурация и релиз-кандидат
## Теория:
* Спецификация OpenAPI (ранее Swagger): как FastAPI под капотом читает твои аннотации типов (Pydantic и зависимости) и автоматически строит интерактивную документацию.
* Культура документации API: почему пустой Swagger — это плохой тон. Как правильно использовать параметры summary, description, tags и response_description внутри декораторов роутера.
* Документирование возможных ошибок: как описать в Swagger, что эндпоинт может вернуть не только 200 OK, но и 400, 401 или 404, используя параметр responses.
* CORS (Cross-Origin Resource Sharing): механизм безопасности браузеров. Почему фронтенд-разработчик не сможет отправить запрос к твоему API, если они запущены на разных портах (например, фронт на localhost:3000, а бэк на localhost:8000), и как настроить CORSMiddleware в FastAPI для решения этой проблемы.
* Валидация переменных окружения: переход от простых словарей и os.getenv к использованию pydantic-settings. Как создать класс Settings, который строго проверяет наличие JWT_SECRET, логинов и паролей от БД при запуске приложения, не давая серверу запуститься, если забыт хотя бы один ключ.
## Практика:
* Внедри CORSMiddleware в main.py. Разреши запросы со всех источников (allow_origins=["*"]) или укажи конкретные порты локалхоста для большей безопасности.
* Отрефактори все эндпоинты твоего Auth-сервиса (регистрация, логин, рефреш токена, получение профиля):
* Сгруппируй их по тегам (например, tags=["Authentication"]).
* Добавь каждому эндпоинту подробное описание (что он делает, какие данные ожидает).
* Пропиши примеры ответов для успешных и ошибочных запросов.
* Создай файл core/config.py и опиши в нем класс конфигурации на базе BaseSettings (из пакета pydantic-settings). Замени все хардкод-строки в проекте (URL базы данных, секретные ключи, время жизни токена) на вызовы из этого класса.
* Открой [http ://127.0.0.1:8000/docs] и проведи полное ручное сквозное (E2E) тестирование своего API прямо из браузера. Твоя документация должна выглядеть профессионально и быть полностью готовой к передаче фронтенд-разработчику.

## Конспект изученных материалов
### 1. Спецификация OpenAPI
* OpenAPI это стандарт для описания REST API, он представляет собой обычный JSON-файл. FastAPI автоматически строит OpenAPI-схему на основе маршрутов, аннотация типов, зависимостей и Pydantic-моделей
### 2. Культура документации API
* Если OpenAPI будет описывать только сам эндпоинт пустой эндпоинт, то это будет непонятно, и поэтому надо дополнять вручную. Для этого используются методы:
    * **summary:** Самое короткое описание, например: *summary='Register User*
    * **description:** Это подробное описание, где можно написать подробности, но, по моему мнению докстринги лучше
    * **tags:** очень полезная вещь, оно группирует эндпоинты по заголовкам, например если есть 100 эндпоинтов, искать среди них в Swagger нужный эндпоинт было бы очень трудным, а с ним можно написать: *tags=['Phones']*. И все эндпоинты с таким тегом будут под этим названием группироваться
    * **response_description:** По умолчанию Swagger пишет: Successful Responce. Но это малоинтформативно, можно написать: Information of user, list of tasks
### 3. Документирование возможных ошибок
* Реализуется с помощью responses:
    * **responses:** это описание ошибок, например эндпоинт получения чего-то может вернуть: 200 OK, но если его нет: 404. Можно описать возможные ответы: *responses={404:{...}, 409:{...}}
### 4. CORS (Cross-Origin Resource Sharing)
* Backend работает на http://localhost:8000 а Frontend на http://localhost:3000. Когда Frontend пытается отправить запрос на сервер, браузер заблокирует этот запрос и выдаст ошибку CORS в консоли. Почему? Из-за политики безопасности браузеров "Same-Origin Policy". Браузер запрещает сайту А делать скрытые запросы к сайту Б, если сайт Б не сказал: Я доверяю сайту А. Иначе любой хакерский сайт мог бы отправлять запросы в любой сервер. CORS даёт разрешение на определённый сайт обращаться к серверу.
### 5. pydantic-settings
* Если читать кофигурацию из файлов .env с помощью os.getenv(...) если в файле нету каких-то данных, то приложение не упадёт сразу же, оно может упасть в любой момент. Используя библиотеку pydantic-settings, описывается класс Settings, в котором строго типизированы секретные данные из .env. Когда FastAPI стартует, Pydantic читает .env файл. Если хотя бы одной переменной не хватает, или вместо числа написан текст, сервер мгновенно выдаст ошибку валидации и не запуститься. Это гарантирует что рабочее приложение всегда имеет 100% правильную конфигурацию.
