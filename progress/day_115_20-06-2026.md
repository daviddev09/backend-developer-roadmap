# День 115 (20.06.2026): Стратегии загрузки данных (Loading Strategies) и борьба с проблемой N+1
## Теория:
* Что такое проблема N+1 запросов в ORM-системах, почему она возникает незаметно для разработчика и моментально «роняет» производительность базы данных под нагрузкой.
* Поведение по умолчанию: Ленивая загрузка (lazy="select"). Что происходит, когда мы запрашиваем список из 100 пользователей, а затем в цикле for обращаемся к user.tasks каждого из них.
* Изучение явных стратегий жадной загрузки (Eager Loading) из модуля sqlalchemy.orm:
* joinedload() — выполняет мгновенный LEFT OUTER JOIN в рамках одного SQL-запроса. Идеально для связей Many-to-One (когда у задачи один юзер). Опасность использования с * коллекциями (создает дубликаты строк в сыром ответе).
* selectinload() — выполняет второй, изолированный запрос с оператором IN (id1, id2, ...). Абсолютный стандарт и лучшая стратегия для загрузки коллекций (One-to-Many и Many-to-Many).
* raiseload() — специальная защитная стратегия, которая выбрасывает ошибку, если код пытается сделать ленивый запрос. Помогает отлавливать неоптимальный код на этапе написания тестов.
## Практика:
* Включи полное логирование SQL-запросов в консоль (параметр echo=True при создании create_engine).
* Напиши функцию, которая выгружает из базы данных всех пользователей, и в цикле выводит на экран названия их задач. Посмотри в консоль и посчитай количество выполненных SQL-запросов (это классический N+1).
* Перепиши этот запрос, применив жадную загрузку: session.scalars(select(User).options(selectinload(User.tasks))).
* Посмотри на логи в консоли снова: убедись, что теперь вместо десятков запросов SQLAlchemy выполняет строго 2 запроса (один для пользователей, один для всех их задач разом).
* Поэкспериментируй со связью задач и тегов, применив цепочку загрузок: options(selectinload(User.tasks).selectinload(Task.tags)).

## Конспект изученных материалов
### 1. Что такое проблема N+1 запросов
* **Своими словами:** Допустим мы хотим вывести список задач всех пользователей. В плохом коде ORM сделает: сначала запрос к БД, чтобы забрать всех пользователей, а потом когда дойдёт до цикла "for task in user.tasks:" ORM будет генерировать запросы Select * from tasks Where user_id = 1, затем user_id=2 и так ещё N запросов, в нашем случае 1+100=101. Первый запрос чтобы получить всех 100 пользователей а последующие 100 запросов чтобы получить задачи каждого по одному, и из-за этого БД ляжет
### 2. Ленивая загрузка (Lazy load)
* **Своими словами:** По умолчанию SQLAlchemy работает по принципу lazy load, он говорит: 'Не стану загружать все данные, пока они не используются', это удобно когда нам надо получить только пользователей, он делает запрос к базе один раз, но когда надо получить задачи пользователей начинается пробелема N+1 загрузки
### 3. Явные стратегии живой загрузки
* **1. joined_load (LEFT OUTER JOIN):** Он делает ровно 1 запрос к БД и удобно для связей Many-to-One, когда надо найти пользователя 50 задач он делает LEFT JOIN и вернёт задачи с пользователем, но нельзя использовать для связей One-To-Many, ведь если у пользователя 50 задач то он вернёт ровно 50 строк в каждой из которых будет дублироваться данные пользователя
* **2. selectinload():** выполняет ровно 2 запроса, в первом запросе получает данные пользователей, а во втором запросе отправляет запрос с id полученных пользователей, например: SELECT * FROM TASKS WHERE user_id IN (1, 2, 3, ...)
* **3. raiseload():** Эта стратегия ничего не загружает, она нужна только для защиты от случайных N+1 запросов в коде. Если кто-то попытается написать user.tasks то БД не станет делать ленивую загрузку а выбросит ошибку
