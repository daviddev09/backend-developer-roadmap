## Конспект изученных материалов
### Что такое Git
* **Git** это система контроаля версий
* Его задача - сохранять историю изменений проекта, чтобы можно было:
    * посмотреть, что изменилось
    * вернуться к любой прошлой версии
    * работать нескольким разработчикам одновременно
    * безопасно экспериментировать
* В Git каждый коммит - полноценная сохраненная версия проекта
### Что такое репозиторий
* **Репозиторий** - это папка проекта, в которой Git хранит историю изменений. Когда делается *git init* Git создаёт скрытую папку: .git и если удалить .git Пше забудет абсолютно всю историю. Сам проект останется. Репозиторий позволяет разработчикам отслеживать правки, сохранять разные версии и безопасно работать в команде.
* Репозитории делятся на два типа:
    * **Локальный** находится прямо в компьютере и изменения видны только владельцу
    * **Удаленный** размещен на онлайн-сервере (например, на Github, Gitlab или Butbucket). Нужен для совместной работы и созданий резервных копий
### Три зоны Git
* **1. Working Directory:** Это обычная папка проекта. В нём даже если меняется например файл: main.py и сохраняется, изменения есть, но Git пока ничего не сохранил. Файл находиться в: Working Directory. То есть даже если файл изменяется, истории Git нет
* **2 Staging Area:** Теперь выполняеться *git add main.py*. И Git говорит: Хорошо. Именно эту версию файла я потом включу в следующий коммит. *git add* не создает коммит, он лишь готовит изменения. Это как собрать документы перед отправкой. **Staging** нуэен для того чтобы если разработчик изменил три файла, но он хочет закоммитить одного, тогда и нужен. Это очень удобно
* **3. Repository:** теперь делается: *git commmit -m '...'*, Git берёт всё содержимое Stagging Area и делает коммит. После этого изменения попадают в .git, то есть становяться частью истории
### Четыре статуса файлов
* **1. Untracked:** Новый файл Git его вообще не знает. Команда: *git status* покажет его как untracked file. Git говорит, Я вижу этот файл но никогда не сохранял.
* **2. Unmodified:** Файл уже есть в последнем коммите, и он не изменялся. Такие файлы не показываются командой: *git status* потому что с ними всё хорошо
* **3. Modified:** Файл уже в Git но внесены изменения. Например было: *print('Hello world')* стало: *print('Hello Git')* Этим Git говорит: Последний коммит содержит старую версию. Исправляется с командой: *git add main.py*
* **4. Staged:** после добавления, Git знает что это именна эта версия попадет в слудеющий коммит, и после коммита файл снова станет **Unmodified**
### Git команды:
* *git status* показывает статус каждого незакоммиченного файла
* *git init* создает новый репозиторий, то есть начинает следить за проектом
* *git add* добавляет изменения в индекс:
    * **Добавить один файл:** *git add main.py*
    * **Несколько:** *git add main.py service.py*
    * **Все:** *git add .*
* *git commit:* сохраняет изменения проекта в истории. **Коммит** это сохраненный снимок проекта. Коммит хранит не просто отдельные файлы, а состояние всего проекта на этот момент. Благодаря этому Git может восстановить прект таким, каким он был в момент создания коммита
* *git log --oneline* показывает историю коммитов, каждая строчка это один коммит
### 1. Ветка и Head
* **Ветка:** это просто крошечный текстовый файл, в котором записан хэш - уникальный код конкретного коммита. Когда добавляется новый коммит, указатель текущей ветки переплывает вперёд, на этот свежий коммит. В отличие от других систем контроля версий, ветка в Git не копирует файлы физически в отдельную папку.
* **Для чего нужен (Своими словами):**
    Изначально создаётся одна ветка, это: main или master, и все остальные ветки ссылаются на него. Например, Я написал телеграм бота, и хочу добавить в него новую фичу, и если это будет занимать много времени, и будет сложно и если есть вероятность сломать работающий код, Я создаю новую ветку. В новой созданной ветке, будут все файлы из main, потом Я беру новый Telegram bot API и в нём тестирую новые функции, а пока Я занимаюсь добавлением новых функций, мой основной бот может работать и обслуживать пользователей. А потом когда закончу, если всё написано успешно, просто сливаю код из новой ветки в ветку main, и удаляю тестовую ветку. А также ветки помогают нескольким разрабам работать вместе, например Я оптимизирую запросы к БД, а другой программист в своей ветке может добавлять кнопки в боте.
* **Head:** это специальный служебный указатель, который отвечает на вопрос: "Где я сейчас нахожусь в репозитории?". Обычно HEAD указывает не напрямую на коммит, а на текущую активную ветку. Когда переключаешься на другую ветку, Git просто берёт указатель и направляет его на имя новой ветки. Файлы в рабочей директории на компьютере мгновенно заменяются на те, которые соответствуют коммиту этой новой ветки. 
### 2. Создание, просмотр и переименование веток
* **Просмотр веток:** *git branch*
    * Чтобы увидеть список всех своих локальных меток. Звездочкой и отлельным цветом будет отмечена ветка, на которой сейчас стоит HEAD (То есть текущая ветка)
    *  **Команда:** *git branch -v* показывает список веток вместе с последним коммитом в каждой из них
* **Создание ветки:** *git branch feature-login*
    * Если написать имя ветки, то эта команда создаёт новую ветку, ярлык указатель который смотрит на тот самый коммит, на котором сейчас находишься, например в main. Команда создания не переключает автоматически на новую ветку
* **Переименование ветки:**
    * Если ветка названа опечаткой, её можно переименовать: *git branch -m <старое имя> <новое имя>*
    * Если уже находишься внутри нужной ветки, то можно просто: *git branch -m <новое имя>*
### 3. Переключение между ветками
* **git checkout:** это команда которая использовалась раньше, она могла одновеременно делать разные вещи: могла переключать ветки, восстанавливать удаленные файлы и откатывать изменения. Чтобы избежать путаницы, разработчики разделили этот функционал на специализированную команду *swith*
* **git switch:**
    * *git switch <имя ветки>* перейти в ветку. Это способ сказать Git: замени мои текущие файлы на файлы из этой ветки и перекинь HEAD туда. Например: *git switch new-branch*
    * *git switch -c <имя ветки>* создать ветку и сразу перейти. Флаг -c (от слова create) создает ветку и моментально переводит HEAD на неё. Например: *git switch -c new-branch*
### 4. Удаление локальных веток
* Когда задача выполнена, и код из ветки уже отправлен и слит с основной веткой, старую локальную ветку нужно удалить, чтобы она не мешалась:
    * *git branch -d <имя ветки>* удалить ветку. Пример: *git branch -d new-branch* Использование строчной буквы -d (от delete) - это безопасное удаление. Git перед удалением проверит, были ли изменения ветки закоммичены или успешно перенесены из него куда-то ещё. Если код написан, но забыт сохранить или слить, Git выдаст ошибку и не даст ветке исчезнуть.
    * *git branch -D <имя ветки>* Принудительное удаление. Если код написан и разработчик понял что он никуда не годится, и он хочет удалить его, проигнорировав предупреждения Git, используется заглавная буква: *git branch -D new-branch*
