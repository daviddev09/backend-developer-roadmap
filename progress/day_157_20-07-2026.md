# День 157 (20.07.2026): Работа с удаленными репозиториями и Pull Requests (PR)
## Теория:
* Отличия локального репозитория от удаленного (remote).
* Понятие origin.
* Что такое Fork (форк) и зачем он нужен в Open Source проектах.
* Жизненный цикл Pull Request в интерфейсе GitHub или GitLab.
## Практика:
* Создай пустой репозиторий на GitHub.
* Привяжи его к локальному репозиторию и сделай push ветки main.
* Создай локально feature-4, сделай коммит и отправь ветку на GitHub (git push -u origin feature-4).
* Зайди на GitHub и нажми кнопку "Compare & pull request". Заполни описание и создай PR.

## Конспект изученных материалов
### 1. Отличия локального репозитория от удаленного
* **Удаленный репозиторий** это версия проекта, которая хранится на сервере в интернете.
### 2. Понятие origin
* Вместо того чтобы каждый раз при отправке кода писать длинный URL-адрес: git push https..., Git позволяет сохранить этот URL под коротким именем. По традиции, самый первый удаленный сервер всегда называют словом origin (источник или начало)
### 3. Что такое Fork
* **Форк** это функция самого GitHub. Когда ты нажимаешь кнопку Fork на странице чужого проекта, GitHub делает точную копию этого репозиторий и кладет её в твой личный аккаунт. И так у тебя есть полные права на эту копию, можно клонировать форк себе на компьютер, создать ветку, писать код, делать коммиты и пушить их обратно в свой форк
### 4. PullRequest в GitHub
* **Pull Request** это специальная страница-обсуждение, где ты говоришь: "Эй, ребята, я тут написал код в своей ветке, посмотрите, пожалуйста, и влейте его в ваш главный проект"
### Команды Git
* *git remote -v* посмотреть список всех удаленных серверов, подключенных к твоему проекту
* *git push <имя удаленного сервера> <имя ветки>* отправить свои локальные коммиты в интернет
* *git pull <имя удаленного сервера> <имя ветки>* скачать чужие свежие коммиты из интернета к себе на компьютер
