# День 136 (04.07.2026): Основы профессионального логирования (модуль logging)
## Теория:
* Почему функция print() запрещена в продакшене: невозможность фильтрации, отсутствие времени выполнения, трудности с записью в файл.
* Уровни логирования (Log Levels): DEBUG (отладка), INFO (информация), WARNING (предупреждение), ERROR (ошибка), CRITICAL (критический сбой).
* Три кита модуля logging:
* * Loggers (Логгеры) — точки входа, куда приложение отправляет сообщения.
* * Handlers (Обработчики) — определяют, куда уйдет лог (в консоль StreamHandler, в файл FileHandler).
* * Formatters (Форматтеры) — определяют внешний вид лога (например: [2026-07-04 14:00:00] [ERROR] Ошибка БД).
## Практика:
* Изучи базовую настройку logging.basicConfig().
* Настрой логирование так, чтобы сообщения уровня INFO и выше выводились в консоль с красивым форматом (Время, Уровень, Сообщение).
* Добавь FileHandler, чтобы сообщения уровня ERROR дополнительно записывались в текстовый файл app_errors.log.
* Напиши скрипт, который искусственно вызывает деление на ноль, перехвати эту ошибку через try/except и запиши её в лог через logger.error("Произошла ошибка", exc_info=True).

## Конспект изученных материалов
### 1. print() запрещена в продакшене
* **Своими словами:** Само собой Python не запрещает использовать print, но в продакшене, где очень много запросов, принт становиться вообще неудобным, нефильтрируемым, а ещё print() выводиться только в консоли и если его закрыть то истории действий тоже исчезнут. Именно поэтому в продакшене print() не используется
### 2. Уровни логирования
* **DEBUG** (10): Детальная информация действий. Например, "Сгенерирован SQL-запрос: SELECT...". Включается только когда разработчик ищет баг
* **INFO** (20): Подтверждение того, что всё работает как ожидалось. Например: "Сервер запущен на порту 8000" и т.п.
* **WARNING** (30): Предупреждение. Произошло что-то странное, но приложение продолжает работу. Например: "Осталось 10% места на диске" и т.п.
* **ERROR** (40): Произошла ошибка, конкретная операция не удалась. Например: "Не удалось сохранить пользователя в БД"
* **CRITICAL** (50): Фатальная ошибка. Программа вообще не может продолжать работу. Например: "База данных удалена"
### 3. Три кита модуля logging
* Архитектура модуля logging гениальна. Она разделена на три независимые сущности:
* **A. Loggers:** Это объекты, через которые приложение создаёт сообщения. Логгер принимает сообщение, проверяет его уровень и если всё ок, передает его дальше-Хэндлерам.
* **B. Handlers:** Это "курьеры". Они решают, куда физически отправится твое сообщение. У одного логгера может быть несколько хэндлеров! Например, один хэндлер может выводить INFO в консоль, а второй может ловить только ERROR и записывать в файл
* **C. Formatters:** Это "стилисты". Они решают, как будет выглядеть твое сообщение. Хэдлер передает сырое сообщение форматтеру, а тот оборачивает его в красивую строку. Для этого используются спциальные переменные джокеры:
* * **%(asctime)s** - точное время
* * **%(name)s** - имя логгера
* * **%(levelname)s** - уровень
* * **%(message)s** - сам текст
