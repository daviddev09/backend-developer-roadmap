# День 147 (08.07.2026): Кэширование на уровне эндпоинтов (Response Caching)
## Теория:
* Паттерн Cache-Aside: стандартный алгоритм работы приложения с кэшем (проверить кэш -> если пусто, сходить в БД -> записать в кэш -> вернуть ответ).
* Кэширование ответов FastAPI: как закэшировать весь HTTP-ответ целиком, чтобы при повторном запросе код эндпоинта и бизнес-логика даже не запускались.
* Библиотека fastapi-cache2: обзор инструмента для декларативного кэширования через декораторы.
* Инвалидация кэша: самая сложная проблема в программировании. Как понять, когда данные в БД изменились и кэш нужно принудительно сбросить, чтобы не показывать пользователю устаревшую информацию.
## Практика:
* Интегрируй fastapi-cache2 в свой проект, инициализировав его в main.py при старте приложения (@app.on_event("startup")), передав ему клиент Redis.
* Найди (или создай) тяжелый GET-эндпоинт (например, получение профиля пользователя с его статистикой), добавь искусственную задержку через asyncio.sleep(2).
* Повесь на эндпоинт декоратор @cache(expire=60).
* Вызови эндпоинт несколько раз. Первый запрос должен занять 2 секунды, а все последующие в течение минуты — несколько миллисекунд.

## Конспект изученных материалов
### 1. Паттерн Cache-Aside
* **Cache-Aside** это когда Redis и Postgres ничего не знают друг о друге. А FastAPI выступает в роли посредника и само решает куда идти в кэш или в БД
* **Алгоритм работы Cache-AsideL**
* * **1.** Приложение получает HTTP-запрос
* * **2.** Приложение формирует уникальный ключ (user_5_profile) и спрашивает Redis: "У тебя есть данные по этому ключу?".
* * **3. Сценарий А (Cache Hit Попадание в кэш):** Redis говорит "ДА" и мгновенно отдает данные. Приложение сразу возвращает их клиенту.
* * **4. Сценарий В (Cache Miss Промах кэша):** Redis отвечает None. Приложение идет делать запрос к Postgres
* * **5.** Получив данные из Postgres, приложение сначала записывает их в Redis, и только потом возвращает ответ клиенту
### 2. Кэширование ответов FastAPI
* **Своими словами:** Если в эндпоинте есть сложное вычисление, то можно кэшировать результат этого эндпоинта. И тогда будет в кэше весь HTTP-ответ эндпоинта и в следующий раз когда придёт запрос, функция эндпоинта не выполниться а сразу же вернётся из кэша, и клиент вместо того чтобы ждать например 2 секунд получает ответ за миллисекунды. Это можно реализовать с помощью библиотеки fastapi-cache2
### 3. Библиотека fastapi-cache2
* Это библиотека, которая делает всё автоматически. Она умеет: Подключиться к Redis, хранить ответы, читать ответы, управлять TTL, создавать ключи. **Главная идея** Кэширование через декоратор: @cache(expire=60) это означает что HHTP-ответ будет храниться 60 секунд.
* **Обзор инструмента и объяснения:**
* * **1. Декоратор: @asynccontextmanager** позволяет использовать конструкцию async with. Функция, обернутая в него, разделяется оператором yield на две части: 1) Код до yield выполняется в момент запуска приложения (on startup), 2) код после yield выполняется в момент остановки приложения (on shutdown)
* * **2. from fastap_cache import FastAPICache:** Это управляющий класс из библиотеки fastapi-cache2. Она создана чтобы автоматически кэшировать оветы от эндпоинтов. Сам по себе FastAPI не умеет кэшировать данные в Redis, а FastAPICache связывает FastAPI, декораторы кэширования и БД (бэкенд) куда будут сохраняться данные. При старте приложения вызывается метод FastAPICache.init(...) в него передается бэкенд и указывается префикс для ключей, чтобы кэш приложения не перемешался с другими данными в Redis. После этой инициализации можно вешать декоратор @cache(expirer=...) на любой эндпоинт
* * **3. from fastapi_cache.backends.redis import RedisBackend:** Это класс прослойка. Библиотека fastapi-cache универсальна. Она может сохранять кэш не только в оперативную память. Но для того чтобы он мог общаться с любой БД нужен единый интерфейс. **RedisBackend** это переводчик, который знает специфику Redis. Он принимает объект подключения Redis: *FastAPICache.init(backend=(RedisBackend(redis), prefix=...)) и объясняет какими именно командами нужно записывать данные в Redis, как их читать и как устанавливать время жизни хэша
* * **4. from redis import asyncio as aioredis:** Это импорт официального асинхронного клиента для работы с Redis. aioredis предоставляет асинциированный клиент. Каждая операция возвращает корутину, которая вызывается через await.
* * **Архитектурная цепочка:* *Декоратор @cache перехватывает запрос -> обращается к FastAPICache -> тот вызывает RedisBackend -> RedisBackend через асинхронный клиент aioredis делает await client.get(...) из Redis. И весь этот процесс безопасно стартует и завершается внутри asynccontextmanager
### 4. Инвалидация кэша
* **Проблема инвалидации:** Если время кэша поставилось на час на эндпоинт /users/me (профиль пользователя):
* * **1.** В 12:00 пользователь запрашивает свой профиль, кэш сохраняет: {'name': 'Daviddev'}.
* * **2.** В 12:05 пользователь меняет имя на 'daviddev09'. БД Postgres обновляется.
* * **3.** В 12:06 пользователь снова вызывает /users/me. И он видит что имя всё ещё не изменилось (будто бы)
* * Потому что кэш будет жить ещё 54 минут. И за всё это время приложение будет отдавать старые данные.
* Есть два пути решения:
* * **Подход 1: Терпимая задержка (Короткий TTL)** Можно просто поставить время кэша на 60 секунд и через столько времени кэш удалиться а пользователь через минуту после обновления профиля получит актуальные данные 
* * **Подход 2: Активная инвалидация (Сброс кэша вручную)** Когда меняются данные в Postgres мы можем с помощью FastAPICache.clear() полностью, либо спуститься на уровень ниже, то есть в слой Сервиса и там вручную удалять по id пользователя и т.п.
