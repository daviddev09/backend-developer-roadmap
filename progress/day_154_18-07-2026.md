## Практика:

* Получи конфликт, открой редактор, выбери нужный вариант, сохрани и заверши merge, сделав коммит.

## Конспект изученных материалов
### 1. Fast-forward слияние
* **Fast-forward** в Git это тип слияния веток, при котором история изменений остается прямой линией. Git просто перемещает указатель текущей ветки вперед до уровня целевой ветки, если в текущей ветке не было новых коммитов с момента их разделения. Дополнительный коммит слияния не создается.
* **Пример:** Есть ветка main который указывает на C3 и мы создаём здесь ветку bugFix а потом делаем коммит и bugFix указывает на C4 а main на C3, и мы просто делаем слияние. Это получиться ровной линией, когда main ещё никто не изменял
### 2. 3-way merge и merge-коммиты
* Когда от main создается новая ветка и начинаются делать изменения в этой новой ветке, кто-то другой может закончить свои изменения в другой ветке который тоже создался от main, и вливается в main история разделяется на два параллельных пути, ведь в main есть изменения. И если попытаться их слить то это уже будет 3-way merge.
* **Как делается 3-way merge:** Когда ты сделаешь git merge feature, находясь в main, Git возьмет три точки отсчета (три коммита):
    * **Target:** Последний коммит вливаемой ветки (feature)
    * **Source:** Последний коммит текущей ветки (main)
    * **Lowecast Common Ancestor/общий предок:** Самый близкий общий родитель обеих веток. Git идет по стрелочкам назад от обоих веток до тех пор, пока пути не пересекутся. 
* **Как происходит само объединение:** Git нужен общий предок (коммит) чтобы понять, кто и что изменил. Git сравнивает файлы не между коммитами веток main и feature, а берет их состояния относительно коммита предка.
* **Merge-коммит** это особый тип коммита в Git, который создается автоматически, когда объединяются две ветки, история которых успела разойтись в разные стороны. Главное отличие merge-коммита от обычного заключается в количестве родителей:
    * Обычный коммит имеет одного родителя
    * Merge-коммит имеет двух (или более) родителей одновременно. Он служит мостом, который соединяет две независимые линии разработки обратно в одну.
### Конфликты слияния
* Конфликт происходит, когда автоматика Git сдается. Например, в ветке main который указывает на C1 относледовался feature и он изменил первую строку одного файла и сделал коммит, в то же время в ветке main изменили первую строку того же файла, и при попытке слить, Git просто не знает какая версия правильнее. В момент конфликта Git отсанавливает процесс слияния и "помечает" проблемные места прямо в коде. Если открыть сломанный файл то можно увидеть подобное:
    * <<<<<<< HEAD
    * Привет, мир!
    * =======
    * Привет, пользователь!
    * *>>>>>>> feature
        * Всё, что находится между <<< HEAD И ====== - это код из той ветки, в которой сейчас находишься
        * Всё, что находится между === и >>>> feature - это код из ветки, которую пытаешься влить
    * Твоя задача - вручную удалить служебные маркеры и оставить только финальный код, который действительно должен пойти в работу
### 4. Инструменты для решения конфликтов
* IDE упрощают процесс ручного слияния. Например в VS Code при открытии конфликтного файла подсвечивает эти блоки разными цветами и предлагает удобные кнопки прямо над кодом.
* После того как выбран файл и сохранен останется лишь завершить слияние:
    * добавить исправленный файл в индекс
    * Завершить процесс: *git commit*
### Команды Git
* *git merge <имя ветки>* - влить в текущую ветку
* *git merge --abort* - отменить процесс слияния при ошибках. Она мгновенно отменит слияние и вернет проект в то состояние, в котором он был до попытки
* *git log --oneline --graph --all* визуализация дерева коммитов в консоли.
